## Confirmed Findings (FAIL)

### 1. CSV Formula Injection (013) - MEDIUM
**File:** `apps/users/views.py:660-727`
**Issue:** CSV export writes user-controlled data (dream/goal/task titles) directly without sanitizing formula characters (`=`, `+`, `-`, `@`). A malicious dream title containing a spreadsheet formula would be written raw to the CSV.
**Mitigation in place:** `Content-Disposition: attachment` forces download (partial mitigation).
**Fix:** Prefix cells starting with `=`, `+`, `-`, `@`, `\t`, `\r` with a single quote `'`.

### 2. Session Not Invalidated on Password Change (040) - HIGH
**File:** `core/auth/serializers.py:192-195` (PasswordChangeSerializer) and `core/auth/serializers.py:159-161` (PasswordResetConfirmSerializer)
**Issue:** `set_password()` + `save()` does not invalidate existing refresh tokens on other devices. An attacker who stole a refresh token retains access even after the victim changes their password. Only account deletion (`apps/users/views.py:468`) properly cleans up `OutstandingToken` records.
**Fix:** After password change/reset, call `OutstandingToken.objects.filter(user=user).delete()` to invalidate all existing tokens, then re-issue tokens for the current session.

### 3. Account Enumeration via Registration (041) - MEDIUM
**File:** `core/auth/serializers.py:67-69`
**Issue:** Registration returns "An account with this email already exists" when email is taken, allowing attackers to enumerate valid email addresses. Login and password reset properly use generic messages.
**Fix:** Return a generic message like "If this email is available, a verification link was sent" and send a notification to the existing account instead.

### 4. OAuth Social Login Missing CSRF Protection (043) - HIGH
**File:** `core/auth/views.py:681-754` (GoogleLoginView), `core/auth/views.py:757+` (AppleLoginView)
**Issue:** Social login endpoints accept `id_token` via POST with `AllowAny` permission and no state/CSRF protection. While an attacker cannot forge a valid Google/Apple `id_token`, a login CSRF attack could force a victim to authenticate as the attacker's account (if the attacker has a valid id_token for their own Google account). This is a login CSRF -- the attacker forces the victim to be logged in as the attacker, potentially capturing the victim's subsequent actions.
**Practical risk:** Low, because the attacker would need to serve their own valid `id_token` to the victim, and the victim would notice they are logged in as someone else. But it violates CSRF best practices.
**Fix:** Add a server-generated nonce/state parameter that must be included in the social login POST.

### 5. Insecure Direct File Reference - Media Files (078) - HIGH
**File:** `config/settings/production.py:93`
**Issue:** `AWS_QUERYSTRING_AUTH = False` generates non-signed S3 URLs for all media files (avatars, dream post images, story images/videos, progress photos, event cover images, vision board images). Anyone with the URL can access these files without authentication. While UUIDs in filenames provide obscurity, URLs shared in browser history, logs, or chat become permanent access links.
**Fix:** Set `AWS_QUERYSTRING_AUTH = True` to generate pre-signed URLs with expiration, or serve media through an authenticated application endpoint.

### 6. Mobile Token Storage in Plain AsyncStorage (097b) - MEDIUM
**File:** `stepora-mobile/src/services/api.js:6,44,83-94`
**Issue:** JWT access and refresh tokens stored in `@react-native-async-storage/async-storage`, which is unencrypted on-device storage (SQLite on Android, plist on iOS). Tokens are readable via device backup extraction, root access, or other apps on rooted/jailbroken devices.
**Fix:** Use `react-native-keychain` (iOS Keychain + Android Keystore) or `expo-secure-store` for token storage.

## Notable Passes (Well-Implemented Security)

1. **Comprehensive rate limiting** with per-endpoint throttle classes, `NUM_PROXIES=1` to prevent XFF spoofing
2. **2FA implementation** with IP-bound challenge tokens, account lockout, narrow TOTP window, hashed backup codes
3. **File upload security** with magic byte validation, content-type checking, UUID filenames, size limits
4. **Security headers** fully configured: CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, COOP, CORP
5. **WebSocket security** with message-based auth (not URL tokens), 10s timeout, rate limiting, heartbeat
6. **XSS prevention** with DOMPurify on all unsafe HTML insertion, CSP with `script-src 'self'`
7. **IDOR prevention** with consistent `get_queryset()` filtering and `perform_create` ownership validation
8. **JWT security** with 15-min access tokens, rotation, blacklisting, secure cookie flags
9. **Password reset** uses hardcoded `FRONTEND_URL` (not Host header), generic response messages
10. **Origin validation middleware** derived from CORS settings, not hardcoded
