## Business Logic (201-230)

### V-201: Price Manipulation in Checkout
**Severity:** CRITICAL | **Result: PASS**

- Server uses Stripe Checkout Sessions for all payments. The `checkout` view in `apps/subscriptions/views.py` resolves the plan server-side via `serializer.get_plan()`, then calls `StripeService.create_checkout_session()` which looks up the `stripe_price_id` from the `SubscriptionPlan` model.
- Client never submits price or amount values. Prices are defined in Stripe and in the DB (`SubscriptionPlan.price_monthly` as `DecimalField`).
- Webhook (`checkout.session.completed`) confirms payment before granting access.
- Stripe verifies the amount matches the Price object.

### V-202: Coupon/Promo Code Abuse (Unlimited Use)
**Severity:** HIGH | **Result: PASS**

- `PromotionRedemption` model has `unique_together = [("promotion", "user")]` preventing multiple redemptions per user per promotion.
- `Promotion.max_redemptions` caps total global uses; `is_exhausted` property checks this.
- Expiration dates (`end_date`) are validated server-side in `PromotionService.get_active_promotions()`.
- `PromotionPlanDiscount` links specific discounts to specific plans preventing stacking.
- Promotion redemption is recorded only in the webhook handler (after payment confirmation), not at checkout creation.
- Rate limiting on subscription endpoints: `"subscription": "5/minute"`.

### V-203: Race Condition in Financial Transaction
**Severity:** CRITICAL | **Result: PASS**

- Financial transactions are delegated to Stripe (checkout sessions, subscription management). No custom balance/wallet system exists.
- `StripeWebhookEvent` model with `stripe_event_id` unique constraint provides idempotency for webhook processing.
- Webhook handler checks `StripeWebhookEvent.objects.filter(stripe_event_id=event_id).exists()` before processing.
- No internal currency/balance system that could suffer from race conditions.

### V-204: Race Condition in Inventory/Stock
**Severity:** HIGH | **Result: N/A**

- The application is a goal-tracking/coaching platform, not an e-commerce platform. No physical inventory or stock system exists.

### V-205: Double-Spending Attack
**Severity:** CRITICAL | **Result: PASS**

- No internal wallet or credit system exists. All payments flow through Stripe.
- Gamification XP uses Redis atomic counters (`F()` expressions) for increments, preventing double-counting.
- Webhook idempotency via `StripeWebhookEvent` prevents duplicate subscription activations.

### V-206: Negative Quantity/Price Exploit
**Severity:** HIGH | **Result: PASS**

- `SubscriptionPlan.price_monthly` is a `DecimalField` with default 0. Plans are admin-created, not user-submitted.
- `PromotionPlanDiscount.discount_value` is admin-managed.
- `discounted_price` property uses `max(0.0, price - value)` preventing negative prices.
- The store app uses similar admin-defined pricing. No user-submitted price values in checkout.

### V-207: Currency Rounding Exploitation
**Severity:** MEDIUM | **Result: PASS**

- `price_monthly` uses `DecimalField(max_digits=6, decimal_places=2)` -- proper fixed-point arithmetic.
- `discount_value` uses `DecimalField(max_digits=8, decimal_places=2)`.
- Actual billing is handled by Stripe which uses integer cents internally.
- `discounted_price` property converts to float only for display, not for charging.

### V-208: Free Trial Abuse (Repeated Signups)
**Severity:** MEDIUM | **Result: FAIL**

- `trial_period_days` is configured per plan, but no mechanism prevents creating multiple accounts for repeat trials.
- No device fingerprinting, no payment method requirement for trial activation.
- No email alias normalization (e.g., stripping `+suffix` from Gmail addresses).
- No disposable email domain blocking.
- **Finding:** A user can create unlimited accounts with different emails to repeatedly abuse free trials.

### V-209: Subscription Downgrade Data Retention
**Severity:** MEDIUM | **Result: PASS**

- Feature access is enforced server-side on every request via subscription-gated permissions (`CanCreateDream`, `CanUseAI`, `CanUseBuddy`, `CanUseCircles`, etc.) in `core/permissions.py`.
- Each permission reads the user's current plan via `request.user.get_active_plan()` on every request.
- Dream creation checks `plan.dream_limit` against current active dream count.
- AI usage quotas are enforced per-day via Redis counters in `core/ai_usage.py`.
- Existing data remains readable after downgrade (read-only access), but new creation is blocked.

### V-210: Referral System Abuse (Self-Referral)
**Severity:** MEDIUM | **Result: FAIL**

- Referral system exists in `apps/referrals/` with `ReferralCode`, `Referral`, and `ReferralReward` models.
- No device fingerprinting or IP correlation to detect self-referrals.
- No check for same IP, device, or payment method between referrer and referred.
- Referral rewards appear to be credited upon signup without requiring meaningful engagement thresholds.
- **Finding:** Users can self-refer by creating accounts with different emails from the same device/IP.

### V-211: Loyalty Points/XP Farming
**Severity:** MEDIUM | **Result: PASS**

- XP is granted server-side only, triggered by task/goal completion events validated by the backend.
- Gamification uses `F()` expressions for atomic increments preventing duplicate counting.
- Daily activity tracking (`DailyActivity` model) provides per-day limits.
- AI usage quotas in `core/ai_usage.py` use Redis atomic counters with daily TTLs.
- Rate limiting on API endpoints prevents rapid automated actions.

### V-212: Gamification Exploit (Artificial Scoring)
**Severity:** MEDIUM | **Result: PASS**

- Scores (XP) are calculated and stored server-side in `GamificationProfile` model.
- No API endpoint accepts client-submitted score values.
- XP is only granted through validated server-side actions (task completion, check-ins, etc.).
- Leaderboards are computed from server-side data in `apps/gamification/views.py`.

### V-213: Feature Gate Bypass (Client-Side Check Only)
**Severity:** HIGH | **Result: PASS**

- All feature gates are enforced server-side via DRF permission classes in `core/permissions.py`.
- `IsPremiumUser`, `IsProUser`, `CanCreateDream`, `CanUseAI`, `CanUseBuddy`, `CanUseCircles`, `CanUseVisionBoard`, `CanUseLeague`, `CanUseStore`, `CanUseSocialFeed`, `CanMakePublicDream` -- all check `request.user.get_active_plan()` on every request.
- Client-side feature flags (`src/config/featureFlags.js` in frontend) are used for UI rendering only.
- API endpoints return 403 with `subscription_required` code when plan check fails.

### V-214: Time-Based Logic Manipulation (Timezone/Clock)
**Severity:** MEDIUM | **Result: PASS**

- Server uses `django.utils.timezone.now()` for all time-based logic (subscription periods, cooldowns, promotion dates).
- Promotion `start_date` and `end_date` are validated server-side against server time.
- AI usage quotas use `date.today()` on the server for daily resets.
- Client-submitted timestamps are not trusted for business logic decisions.

### V-215: Workflow Skip (Jump Steps in Multi-Step Process)
**Severity:** HIGH | **Result: PASS**

- Subscription checkout flows through Stripe Checkout Sessions -- steps cannot be skipped.
- Email verification is enforced via `IsEmailVerified` permission class on all API endpoints (except auth endpoints and /users/me/).
- Dream calibration flow tracks state server-side.
- Onboarding completion is tracked server-side on the user model.

### V-216: Parallel Request Resource Exhaustion
**Severity:** HIGH | **Result: PASS**

- Connection pooling configured: `CONN_MAX_AGE = 600` with `connect_timeout: 10` and `statement_timeout=30000` (30s) in production settings.
- PgBouncer sidecar in ECS handles connection pooling (pool_mode=transaction).
- ScopedRateThrottle is configured with per-scope limits (auth: 5/min, ai_chat: 10/min, etc.).
- `NUM_PROXIES: 1` prevents throttle bypass via spoofed X-Forwarded-For.
- Celery handles long-running tasks asynchronously.

### V-217: Idempotency Violation (Duplicate Operations)
**Severity:** HIGH | **Result: PASS (Partial)**

- Stripe webhook processing has idempotency via `StripeWebhookEvent` model with unique `stripe_event_id`.
- Subscription endpoints have rate limiting (5/min).
- `PromotionRedemption` has `unique_together` constraint preventing duplicate redemptions.
- **Note:** General API endpoints (dream creation, task updates) lack idempotency keys, but these are not financial operations and duplicate creates are unlikely to cause harm beyond duplicate records.

### V-218: Order State Manipulation
**Severity:** HIGH | **Result: PASS**

- Subscription status transitions are managed entirely by Stripe webhooks, not user API calls.
- The `change_plan`, `cancel`, and `reactivate` actions go through `StripeService` which calls the Stripe API.
- Users cannot directly set their subscription status via API.
- Dream/goal status changes are validated server-side with ownership checks.

### V-219: Cart Abandonment Exploit
**Severity:** LOW | **Result: N/A**

- No shopping cart system exists. Subscriptions use Stripe Checkout Sessions (ephemeral, expire automatically).

### V-220: Insufficient Anti-Automation (Bot Abuse)
**Severity:** HIGH | **Result: FAIL**

- No CAPTCHA or equivalent challenge on signup, login, or checkout.
- Rate limiting exists (auth: 5/min, register: 5/min) but is the only anti-bot defense.
- No device fingerprinting, no behavioral analysis, no honeypot fields.
- No headless browser detection.
- **Finding:** Automated scripts can create accounts and perform actions at scale, limited only by rate limiting.

### V-221: Account Takeover via Social Engineering
**Severity:** HIGH | **Result: N/A (Partial)**

- No customer support portal or agent system exists in the codebase.
- Password reset requires email confirmation.
- Email change requires current password verification (`check_password`).
- No MFA bypass path for support agents.
- However, no cooling-off period after email/password changes.

### V-222: Account Takeover via SIM Swap
**Severity:** CRITICAL | **Result: N/A**

- The application does not use SMS-based MFA or SMS for password recovery.
- Authentication is email/password + optional TOTP 2FA (`apps/users/two_factor.py`).
- No phone number is required for authentication.

### V-223: Insufficient Verification Before Critical Action
**Severity:** HIGH | **Result: PASS**

- Account deletion (`delete_account`) requires password re-entry: `user.check_password(password)`.
- Password change requires `old_password` validation (in `core/auth/serializers.py`).
- 2FA setup/disable requires password verification (`apps/users/two_factor.py` line 224, 315).
- Data export would need similar verification but no export endpoint found.

### V-224: Missing Re-Authentication for Sensitive Operations
**Severity:** HIGH | **Result: PASS**

- Password change requires current password (`old_password` field in serializer).
- Account deletion requires password.
- 2FA enable/disable requires password.
- Email change in profile requires password verification at `apps/users/views.py:761`.

### V-225: Business Logic in Client-Side Code
**Severity:** HIGH | **Result: PASS**

- All business rules enforced server-side: subscription plan checks, dream limits, AI quotas, feature gates.
- Pricing defined in DB and Stripe, never in client code.
- Frontend code (`featureFlags.js`) only controls UI visibility, not access.
- Server validates all input independently.

### V-226: Insecure Direct Download Link
**Severity:** MEDIUM | **Result: PASS**

- Media files use S3 with `AWS_QUERYSTRING_AUTH = False` (direct URLs, not pre-signed).
- However, `core/storage.py` provides a `presigned_url()` function for authenticated access to sensitive files.
- Media uploads use UUIDs in filenames (not sequential IDs).
- Vision board and avatar files are intended to be semi-public (shared within the app).

### V-227: Quota Bypass (Resource Limits)
**Severity:** MEDIUM | **Result: PASS**

- AI usage quotas enforced per-account via Redis atomic counters in `core/ai_usage.py`.
- Dream limits checked per-user in `CanCreateDream` permission.
- Quotas reset daily using server-side date, not client-submitted values.
- Redis TTL ensures automatic key expiration.

### V-228: Notification/Email Flooding
**Severity:** MEDIUM | **Result: FAIL**

- Password reset and verification emails have rate limiting via `AuthPasswordRateThrottle` (5/min).
- However, actions that trigger notifications (follows, messages, reactions) lack per-recipient rate limits.
- No notification deduplication for identical events within a time window.
- **Finding:** A user could generate excessive notifications to a target by rapidly performing social actions (follow/unfollow, send messages).

### V-229: Content Spam/Abuse Without Moderation
**Severity:** MEDIUM | **Result: PASS (Partial)**

- User reporting system exists (`ReportedUser` model, `report_user` endpoint in `apps/social/views.py`).
- Rate limiting on content creation endpoints.
- No automated content filtering for spam, profanity, or malicious URLs.
- No moderation queue for new-user content.
- **Note:** Basic infrastructure exists (reporting) but automated filtering is absent.

### V-230: Review/Rating Manipulation
**Severity:** MEDIUM | **Result: N/A**

- No review or rating system exists in the application. The app is a goal-tracking platform, not a marketplace.

## Mobile Security (231-265)

### V-231: Insecure Data Storage (Plaintext on Device)
**Severity:** HIGH | **Result: FAIL**

- Authentication tokens (access and refresh) are stored in `AsyncStorage` (`dp-access-token`, `dp-refresh-token`) as seen in `src/services/api.js`.
- `AsyncStorage` is unencrypted on both Android and iOS.
- No use of `react-native-keychain` or `react-native-encrypted-storage` for token storage.
- Biometric settings also stored in `AsyncStorage` (non-sensitive, acceptable).
- **Finding:** JWT tokens stored in plaintext AsyncStorage. Should use Keychain (iOS) / Keystore (Android) via `react-native-keychain`.

### V-232: Missing Root/Jailbreak Detection
**Severity:** MEDIUM | **Result: FAIL**

- No root/jailbreak detection code found in the mobile app source.
- No SafetyNet/Play Integrity (Android) or DeviceCheck (iOS) integration.
- No detection of Magisk, Cydia, or other root/jailbreak indicators.
- **Finding:** App runs without restriction on rooted/jailbroken devices.

### V-233: Insufficient Transport Layer Security (Mobile)
**Severity:** HIGH | **Result: PASS**

- Android `network_security_config.xml` sets `cleartextTrafficPermitted="false"` globally.
- AndroidManifest.xml has `android:usesCleartextTraffic="false"`.
- iOS Info.plist sets `NSAllowsArbitraryLoads` to `false` with specific domain configs that also disallow insecure HTTP.
- API URL configured as `https://api.stepora.app`.
- WebSocket URL configured as `wss://api.stepora.app`.

### V-234: Missing Certificate Pinning
**Severity:** HIGH | **Result: FAIL**

- No certificate pinning configured in `network_security_config.xml` (no `<pin-set>` element).
- No pinning in iOS configuration.
- No pinning at the HTTP client level (no OkHttp interceptors for pinning).
- `network_security_config.xml` only trusts system certificates.
- **Finding:** MITM possible with an installed proxy CA certificate.

### V-235: Insecure IPC (Intent/URL Scheme Hijacking)
**Severity:** HIGH | **Result: PASS (Partial)**

- MainActivity is `exported="true"` (required for launcher).
- Firebase service is `exported="false"`.
- Custom URL scheme `com.stepora.app` uses reverse-domain naming (good).
- Universal Links configured with `android:autoVerify="true"`.
- **Minor concern:** Intent data from deep links should be validated before use.

### V-236: Deep Link Hijacking
**Severity:** HIGH | **Result: PASS (Partial)**

- Android App Links configured with `autoVerify="true"` for `stepora.app` domain.
- Custom URL scheme uses reverse-domain naming (`com.stepora.app`).
- iOS Universal Links configured in entitlements (referenced but not seen inline).
- OAuth redirect for Stripe uses custom scheme (`com.stepora.app://subscription/success`).
- **Note:** The custom scheme for Stripe redirects could theoretically be hijacked, but App Links with verification mitigate the primary risk.

### V-237: Clipboard Data Leakage
**Severity:** MEDIUM | **Result: N/A**

- No sensitive data copying features found in the mobile app source code.
- Clipboard usage limited to sharing referral links and social feed content.
- No copying of passwords, tokens, or financial data to clipboard.

### V-238: Keyboard Cache Sensitive Data

### V-239: Screenshot/Screen Recording Not Blocked
**Severity:** MEDIUM | **Result: FAIL**

- No `FLAG_SECURE` usage found in Android code.

### V-240: Debug Logs in Production Build

## Infrastructure (266-300)

### V-266: Docker Container Running as Root
**Severity:** HIGH | **Result: PASS**

- Dockerfile creates a dedicated `appuser` (UID 1000) and switches to it via `USER appuser` before CMD.

**Severity:** HIGH | **Result: PASS (Partial)**

- `docker-compose.yml` references `${DB_PASSWORD}`, `${REDIS_PASSWORD}`, `${ELASTIC_PASSWORD}` from environment.
- **Note:** Secrets are still passed as environment variables to containers (visible in `docker inspect` / `/proc/1/environ`), but this is the standard ECS pattern. AWS Secrets Manager integration is the recommended approach for ECS.
