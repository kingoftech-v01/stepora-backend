# Security Audit Report: Vulnerabilities 301-400

**Date:** 2026-03-26
**Repos audited:** `/root/stepora` (backend), `/root/stepora-frontend` (frontend), `/root/stepora-mobile` (mobile)
**Auditor:** Automated code review (report only, no fixes applied)

**Legend:** PASS = mitigated, FAIL = vulnerable/missing, N/A = not applicable to this codebase

## Denial of Service (301-330)

### V-301: Regular Expression DoS (ReDoS) -- PASS
**Severity:** HIGH
- All regex patterns in `core/validators.py` and `core/ai_validators.py` use simple, anchored patterns (e.g., `^\d{4}-\d{2}-\d{2}$`) without nested quantifiers.
- `core/moderation.py` uses compiled regex patterns with alternation but no nested quantifiers that would cause catastrophic backtracking.
- No user-supplied regex patterns are compiled or executed.

### V-302: Hash Collision DoS -- PASS
**Severity:** HIGH
- Python 3.3+ uses randomized hash seeds by default (PYTHONHASHSEED is random unless overridden).
- Django's `DATA_UPLOAD_MAX_MEMORY_SIZE` is set to ~110 MB in `config/settings/base.py:201`.
- POST parameter counts are bounded by Django's defaults (`DATA_UPLOAD_MAX_NUMBER_FIELDS = 1000`).

### V-303: XML Bomb (Billion Laughs) -- N/A
**Severity:** CRITICAL
- The application does not accept XML input. No XML parsing libraries (lxml, defusedxml, etree) are imported anywhere in the backend codebase.
- All API endpoints use JSON. No SOAP or XML endpoints exist.

### V-304: JSON Bomb (Deeply Nested) -- FAIL
**Severity:** HIGH
- No explicit JSON nesting depth limit is configured in Django or DRF settings.
- `DATA_UPLOAD_MAX_MEMORY_SIZE` is set to ~110 MB which limits total payload size but does not prevent deeply nested structures.
- Python's default recursion limit (1000) provides some implicit protection, but no explicit depth limit is enforced at the parser level.

### V-305: Zip Bomb -- N/A
**Severity:** HIGH
- The application does not accept or decompress archive files (ZIP, GZIP, TAR).
- OTA bundles reference `.zip` in comments/docs (`core/storage.py`) but the storage class handles pre-signed URLs, not server-side decompression.

### V-306: Slowloris Attack -- PASS
**Severity:** HIGH
- Production uses daphne (ASGI) behind AWS ALB. ALB has built-in idle connection timeout (default 60s).
- ALB terminates slow connections before they reach the application.
- ECS Fargate provides network-level isolation.

### V-307: HTTP/2 Rapid Reset -- PASS
**Severity:** CRITICAL
- AWS ALB handles HTTP/2 termination and is managed/patched by AWS. AWS patched CVE-2023-44487 in October 2023.
- Application servers (daphne) receive HTTP/1.1 from ALB.

### V-308: Application-Layer DDoS -- PASS (partial)
**Severity:** CRITICAL
- Rate limiting is implemented across all public endpoints via DRF throttles (`config/settings/base.py:251-273`).
- Auth endpoints: 5/minute, AI endpoints: 5-15/minute, search: 15/minute.
- AWS ALB provides basic protection. However, no dedicated WAF or DDoS mitigation (AWS Shield Advanced, Cloudflare proxy) is deployed.

### V-309: Resource Exhaustion via Large File Upload -- PASS
**Severity:** HIGH
- `DATA_UPLOAD_MAX_MEMORY_SIZE = 115_343_360` (~110 MB) in `config/settings/base.py:201`.
- `FILE_UPLOAD_MAX_MEMORY_SIZE = 10_485_760` (10 MB, files above go to disk).
- Upload throttle: 10/minute. Media files stored on S3 (not local disk).

### V-310: Resource Exhaustion via Unbounded Query -- PASS
**Severity:** HIGH
- Mandatory pagination on all list endpoints: `DEFAULT_PAGINATION_CLASS = StandardLimitOffsetPagination` with `max_limit = 100`.
- `StandardResultsSetPagination` has `max_page_size = 100`; `LargeResultsSetPagination` has `max_page_size = 200`.
- Database statement timeout of 30 seconds in production (`config/settings/production.py:192`).

### V-311: Recursive Serializer (Infinite Nesting) -- PASS
**Severity:** HIGH
- No `Meta.depth` is set on any serializer (verified via grep). Serializers use explicit field declarations and flat ID references for relationships.
- Nested serializers are used for detail views only with controlled nesting (e.g., `select_related` with specific fields).
- No circular serializer references detected.

### V-312: N+1 Query Performance DoS -- PASS (partial)
**Severity:** MEDIUM
- `select_related` and `prefetch_related` are used extensively across viewsets (30+ usage sites found in views, services, and serializers).
- Pagination limits (max 100-200 items) bound worst-case query counts.
- However, there is no automated N+1 detection in tests or CI/CD pipeline (e.g., `nplusone` or query count assertions).

### V-313: Cache Stampede -- FAIL
**Severity:** MEDIUM
- Redis caching is used for sessions and AI usage tracking. No cache lock/mutex patterns, stale-while-revalidate, or probabilistic early expiration are implemented.
- No explicit cache stampede protection found in the codebase.

### V-314: Connection Pool Exhaustion -- PASS
**Severity:** HIGH
- PgBouncer sidecar is deployed in each ECS task (backend, celery, celery-beat) in transaction pooling mode.
- `CONN_MAX_AGE = 600` in production settings.
- Database connect timeout of 10 seconds and statement timeout of 30 seconds configured.

### V-315: Thread Pool Exhaustion -- PASS
**Severity:** HIGH
- Production uses daphne (ASGI) which is asynchronous by nature.
- Heavy operations (AI, email, reports) are offloaded to Celery background workers.
- External API calls (OpenAI, Stripe, Agora) are made from Celery tasks or have timeouts.

### V-316: Disk Space Exhaustion (Log Flooding) -- PASS
**Severity:** MEDIUM
- Production logging is stdout-only (ECS CloudWatch), no local disk files.
- Root log level is WARNING in production (`config/settings/production.py:144`).
- CloudWatch log retention is 30 days.

### V-317: Memory Exhaustion (Large Payload Processing) -- PASS (partial)
**Severity:** HIGH
- Request body size limited to ~110 MB. Files above 10 MB go to disk temp storage.
- ECS Fargate containers have memory limits (512 MB, based on task definitions).
- However, no streaming JSON parser is used -- DRF parses entire JSON bodies into memory.

### V-318: CPU Exhaustion (Crypto/Hash Operations) -- PASS
**Severity:** HIGH
- All auth endpoints have strict rate limits: login 5/minute, register 5/minute, password reset 5/minute.
- Account lockout after repeated failed attempts (detected in `core/auth/views.py`).
- `EmailVerificationRateThrottle`: 10/hour.

### V-319: WebSocket Connection Flooding -- FAIL
**Severity:** HIGH
- WebSocket authentication is required (`core/websocket_auth.py` validates tokens).
- However, no per-IP or per-user WebSocket connection limits are configured.
- No idle WebSocket timeout or connection rate limiting found in the WebSocket middleware.

### V-320: Email Queue Flooding -- PASS
**Severity:** MEDIUM
- Email-triggering endpoints have rate limits: auth 5/minute, email verification 10/hour.
- Password reset, verification, and registration all use throttle classes.
- Emails are sent asynchronously via Celery tasks.

### V-321: Push Notification Flooding -- PASS (partial)
**Severity:** MEDIUM
- Push notifications are sent via Celery tasks (FCM service in `apps/notifications/fcm_service.py`).
- Some notification triggers have rate limits (e.g., `ai_motivation` 5/day).
- However, no per-recipient notification rate limiting or deduplication is explicitly implemented.

### V-322: API Pagination Abuse (page=999999) -- PASS
**Severity:** MEDIUM
- Default pagination uses `LimitOffsetPagination` with `max_limit = 100`. Very large offsets still require DB scanning.
- Database statement timeout of 30 seconds provides a backstop.
- `validate_pagination_params` in `core/validators.py` validates page and page_size parameters.

### V-323: Search Query DoS -- PASS
**Severity:** HIGH
- `validate_search_query` function in `core/validators.py` sanitizes and limits search input length.
- Search endpoints have rate limits (15/minute).
- Elasticsearch is disabled in production (`ELASTICSEARCH_DSL_AUTOSYNC = False`), search uses Django ORM with database-level query timeouts.

### V-324: Report Generation DoS -- PASS
**Severity:** MEDIUM
- Weekly reports are generated asynchronously via Celery tasks (`apps/users/tasks.py`).
- Data export is rate limited to 1/day (`ExportRateThrottle`).
- AI-powered report generation happens in background workers with timeout controls.

### V-325: File Processing DoS (Large Image/Video) -- FAIL
**Severity:** HIGH
- `ImageField` is used for avatars and social content uploads, but no explicit image dimension validation (max pixel count) is configured.
- No server-side image resizing or thumbnail generation with dimension limits found.
- Django `ImageField` validates that the file is a valid image but does not check pixel dimensions against decompression bomb limits.

### V-326: Cascading Failure (No Circuit Breaker) -- FAIL
**Severity:** HIGH
- No circuit breaker pattern is implemented for external service calls (OpenAI, Stripe, Agora, Google Calendar).
- Celery tasks have retry logic with `max_retries` but no circuit breaker to stop calling a failing service.
- If OpenAI or Stripe goes down, requests continue to pile up and timeout.

### V-327: Database Lock Contention -- PASS (partial)
**Severity:** MEDIUM
- No `SELECT FOR UPDATE` or explicit locking patterns found in the codebase.
- Most operations use simple ORM updates which acquire row-level locks briefly.
- PgBouncer with transaction pooling helps manage connection-level contention.
- No explicit lock wait timeout configuration beyond the 30s statement timeout.

### V-328: Distributed Cache Failure -- FAIL
**Severity:** HIGH
- Redis is used for caching, sessions, Celery broker, and channel layers.
- No graceful degradation or fallback is implemented when Redis is unavailable.
- Single Redis instance (ElastiCache `cache.t3.micro`) with no replication or sentinel configuration.

### V-329: Rate Limit Bypass via IP Rotation -- PASS (partial)
**Severity:** MEDIUM
- Auth throttles use `AnonRateThrottle` (IP-based) and `UserRateThrottle` (user-based) for authenticated endpoints.
- Multi-dimensional rate limiting exists (both IP and user-level).
- However, unauthenticated endpoints (login, register) rely solely on IP-based rate limiting, which can be bypassed with IP rotation.

### V-330: Rate Limit Bypass via Header Spoofing -- PASS
**Severity:** HIGH
- `NUM_PROXIES = 1` is configured in DRF settings (`config/settings/base.py:250`), telling DRF to trust only the last X-Forwarded-For entry.
- `_get_client_ip` in `core/auth/views.py:85-88` uses `xff.split(",")[-1].strip()` (last entry, appended by ALB) rather than the first (user-controlled).
- This prevents X-Forwarded-For spoofing to bypass rate limits.

## Privacy & Compliance (331-360)

### V-331: GDPR Right to Erasure Not Implemented -- PASS
**Severity:** CRITICAL
- Self-service account deletion via `POST /api/v1/users/delete-account/` (`apps/users/views.py:396-492`).
- Soft-delete with 30-day grace period, then hard-delete via scheduled Celery task (`hard_delete_expired_accounts`).
- Personal data is anonymized immediately on soft-delete (email, display_name, bio, location, avatar, social_links, preferences).
- Stripe subscriptions cancelled, buddy pairings ended, circle memberships removed.
- JWT tokens and DRF tokens deleted on account deletion.

### V-332: GDPR Right to Data Portability Missing -- PASS
**Severity:** HIGH
- Data export endpoint: `GET /api/v1/users/export-data/` (`apps/users/views.py:510-534`).
- Supports JSON and CSV formats via `?export_format=csv`.
- Exports include profile, dreams, conversations, achievements, focus sessions, notifications.
- Rate limited to 1/day. Audit logged via `log_data_export`.

### V-333: GDPR Consent Not Recorded -- FAIL
**Severity:** HIGH
- Registration requires agreement to terms (`agreedToTerms` validation in frontend).
- However, no backend model or table records the consent with timestamp, scope, and version.
- No mechanism to track consent withdrawal or version history.

### V-334: GDPR Data Processing Without Legal Basis -- FAIL
**Severity:** CRITICAL
- No Record of Processing Activities (ROPA) exists in the codebase or documentation.
- No documented legal basis for each processing activity.
- This is a documentation/compliance gap rather than a code issue.

### V-335: Privacy Policy Outdated/Missing -- PASS
**Severity:** HIGH
- Privacy policy page exists in the site vitrine (`stepora.net/privacy/`).
- Frontend routes to `/privacy` and `/terms` pages (`App.jsx:567-568`).
- Registration screen links to both terms and privacy policy.

### V-336: Cookie Consent Not Implemented -- FAIL
**Severity:** HIGH
- No cookie consent banner or consent management platform is implemented.
- The frontend uses httpOnly cookies for JWT refresh tokens (essential for authentication) but no cookie consent UI exists.
- Google Fonts are loaded on every page without consent.

### V-337: Third-Party Tracking Without Consent -- PASS
**Severity:** HIGH
- No Google Analytics, Facebook Pixel, or other third-party tracking scripts are loaded in the frontend.
- The only third-party scripts are Google Fonts (CSS) and Agora SDK (loaded on-demand for calls).

### V-338: Analytics Collecting PII -- N/A
**Severity:** HIGH
- No analytics platform (Google Analytics, Mixpanel, etc.) is integrated in the frontend or backend.

### V-339: User Data Shared with Third Parties Without Consent -- PASS (partial)
**Severity:** CRITICAL
- User data is shared with: OpenAI (AI features), Stripe (payments), Agora (calls), Google Calendar (calendar sync), Firebase (push notifications).
- All are functional integrations, not data sales. Data minimization is practiced (only necessary data sent).
- However, the privacy policy should document all these third-party processors explicitly. No DPA evidence in the codebase.

### V-340: Data Retention Policy Not Enforced -- FAIL
**Severity:** HIGH
- Account deletion has a 30-day soft-delete + hard-delete schedule.
- However, no general data retention policies or automated purging exist for other data categories (old notifications, chat messages, completed dreams, logs).
- CloudWatch logs have 30-day retention, but no DB-level retention for historical data.

### V-341: Account Deletion Not Complete (Residual Data) -- PASS (partial)
**Severity:** HIGH
- Soft-delete anonymizes personal fields (email, name, bio, location, avatar, social_links, preferences).
- Hard-delete uses Django CASCADE, which deletes all related data in the DB.
- However, residual data may persist in: Redis caches, Elasticsearch indices (if enabled), third-party services (Stripe, OpenAI, Firebase), and CloudWatch logs.

### V-342: CCPA Do Not Sell Not Implemented -- FAIL
**Severity:** HIGH
- No "Do Not Sell My Personal Information" link or mechanism exists.
- No Global Privacy Control (GPC) signal handling.
- If the app targets California users, CCPA compliance is required.

### V-343: Children's Data Collection (COPPA) -- FAIL
**Severity:** CRITICAL
- No age verification during registration. No date of birth field or age gate.
- No parental consent mechanism for users under 13.
- The app collects personal data (email, display name, location, dreams/goals) that would be covered by COPPA for minors.

### V-344: Cross-Border Data Transfer Without Safeguards -- FAIL
**Severity:** HIGH
- AWS infrastructure is in eu-west-3 (Paris), so EU data stays in EU.
- However, data is sent to US-based services (OpenAI, Stripe, Agora, Firebase/Google) without documented SCCs or Transfer Impact Assessments.

### V-345: Data Breach Notification Process Missing -- FAIL
**Severity:** CRITICAL
- No documented data breach response plan, notification templates, or incident response procedures found in the codebase or documentation.
- Sentry is configured for error tracking but not specifically for breach detection.

### V-346: Privacy by Design Not Applied -- PASS (partial)
**Severity:** MEDIUM
- Data minimization is partially practiced (API responses return only necessary fields via serializers).
- Privacy-protective defaults exist (e.g., private profiles by default based on serializer fields).
- However, no formal Privacy Impact Assessments or DPIAs are documented for features.

### V-347: Data Minimization Not Practiced -- PASS (partial)

### V-355: Pixel Tracking in Emails -- PASS

- Email templates use Django template rendering (HTML emails via Celery tasks).
- No tracking pixel or open-tracking mechanism found in email templates.

### V-356: Shadow Profiles (Data from Non-Users) -- PASS

## Frontend Security (361-400)

### V-361: Robots.txt Exposing Private Paths -- PASS

### V-362: Sitemap Exposing Private URLs -- N/A

### V-363: Meta Tags Leaking Private Info -- PASS
**Severity:** MEDIUM
- `index.html` contains only generic meta tags (theme-color, apple-mobile-web-app-title).

### V-364: Social Sharing Metadata Leak (og:tags) -- PASS
**Severity:** MEDIUM
- No Open Graph or Twitter Card meta tags found in `index.html` or the SPA.

### V-365: WHOIS Privacy Not Enabled -- N/A (cannot assess)
**Severity:** LOW

### V-366: Prototype Pollution -- PASS
**Severity:** HIGH
- No deep merge or recursive object assignment on user-controlled input found.

### V-367: Client-Side Path Traversal -- PASS
**Severity:** MEDIUM
- API URL construction uses predefined endpoint constants from `endpoints.js`, not user input in path segments.

### V-368: PostMessage Origin Not Validated -- PASS (partial)
**Severity:** HIGH

- However, the Service Worker message handler does not explicitly validate `event.origin`.

### V-369: Web Worker Injection -- N/A
**Severity:** HIGH
- No `new Worker()` or `new SharedWorker()` calls in the codebase.
- SharedWorker is mentioned only in a TODO file.

### V-370: Service Worker Cache Poisoning -- PASS (partial)
**Severity:** HIGH
- Service Worker is registered in `main.jsx` for push notifications.

- However, no explicit cache versioning/purging or integrity validation in the SW found in the frontend source.
- Service Worker scope is not explicitly limited.

### V-371: IndexedDB Data Not Encrypted -- N/A
**Severity:** MEDIUM
- No IndexedDB usage found in the frontend or mobile codebase.

### V-372: LocalStorage Sensitive Data -- PASS
**Severity:** HIGH
- **Frontend (web):** Access token is stored in memory only (`_accessToken` variable in `api.js:16`). Refresh token uses httpOnly cookie (not localStorage). Legacy `dp-token` in localStorage is migrated and removed on first use.

### V-372-mobile: Mobile Token Storage -- FAIL
**Severity:** HIGH
- React Native mobile app stores access and refresh tokens in `AsyncStorage` (`src/services/api.js:83-94`), which is unencrypted plain text storage.

### V-373: SessionStorage Misuse -- PASS
**Severity:** MEDIUM
- No `sessionStorage` usage found in either frontend or mobile codebase.

### V-374: Insecure iframe Embedding -- PASS
**Severity:** HIGH
- No iframe elements found in the frontend codebase.
- Stripe Checkout uses redirect-based flow, not embedded iframe.

### V-375: Clickjacking via Missing X-Frame-Options -- PASS
**Severity:** HIGH

### V-376: Formjacking (Injected Payment Skimmer) -- PASS
**Severity:** CRITICAL
- Stripe Checkout is handled via server-side session creation and redirect (not client-side card form).

### V-377: DOM Clobbering -- PASS
**Severity:** MEDIUM

### V-378: Dangling Markup Injection -- PASS
**Severity:** MEDIUM

### V-379: CSS Injection (Data Exfiltration) -- PASS

- No user-controlled CSS injection points found. All styles are inline or CSS-in-JS (CSS variables).

### V-380: Font Injection -- PASS (partial)

**Severity:** MEDIUM

### V-383: CDN Compromise Risk -- PASS
