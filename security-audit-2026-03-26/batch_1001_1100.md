## Database Security (1001-1040)

### 1001 - Database User with DBA Privileges
**Severity:** CRITICAL | **Result: PASS**

- Backend uses env vars `DB_USER` / `DB_PASSWORD` from AWS Secrets Manager (`config/settings/base.py:131-136`).
- Production connects through PgBouncer sidecar (`localhost:6432`) to RDS.
- RDS user `stepora` is a dedicated non-superuser per the infrastructure setup (per `AWS_INFRASTRUCTURE.md`).
- Django ORM only needs SELECT/INSERT/UPDATE/DELETE + migration DDL, consistent with a non-DBA user.
- **Note:** The same user is used for both runtime and migrations (no separate migration-only account). Low risk since migration runs at deploy time in the same container.

### 1002 - Database Without Encryption at Rest
**Severity:** HIGH | **Result: PASS**

- Database is AWS RDS PostgreSQL 15 (`stepora-db.c94aou6wywvf.eu-west-3.rds.amazonaws.com`).
- AWS RDS enables encryption at rest by default with AWS-managed KMS keys.
- Keys managed by AWS KMS, not stored alongside data.

### 1003 - Database Without Encryption in Transit
**Severity:** HIGH | **Result: PASS**

- Production settings enforce SSL: `sslmode = os.getenv("DB_SSLMODE", "require")` (`config/settings/production.py:193`).
- PgBouncer sidecar connects to RDS over TLS; backend connects to PgBouncer on localhost (acceptable since same task/container).
- AWS RDS PostgreSQL enforces TLS by default.

### 1004 - Database Backup Not Encrypted
**Severity:** HIGH | **Result: PASS**

- AWS RDS automated backups inherit the encryption-at-rest setting from the database instance.
- Since the RDS instance has encryption enabled, backups are automatically encrypted with the same KMS key.

### 1005 - Database Backup Publicly Accessible
**Severity:** CRITICAL | **Result: PASS**

- RDS instance is in a private subnet (VPC `vpc-07d8842dc93e6d3c2`).
- RDS automated backups and snapshots are stored within AWS infrastructure and are not publicly accessible by default.
- No S3 buckets containing database dumps were found in the codebase.

### 1006 - Database Connection String Hardcoded
**Severity:** HIGH | **Result: FAIL**

- Production credentials are properly loaded from env vars / AWS Secrets Manager.
- **However**, `test_e2e_dual_dreams.py:19` contains `os.environ["DB_PASSWORD"] = "stepora_dev_password"` -- a hardcoded dev password in test code.
- `.gitignore` excludes `.env` files (line 20-22), which is good.
- `Dockerfile:60` contains a hardcoded placeholder `FIELD_ENCRYPTION_KEY` for build-time `collectstatic` only. While documented as a placeholder, it is visible in the Docker image layers.

### 1007 - Database Password Weak/Default
**Severity:** CRITICAL | **Result: PASS**

- Production password is stored in AWS Secrets Manager and injected at runtime.
- Dev password in test code (`stepora_dev_password`) is only used in local dev/test.
- No default passwords are used in production settings; `DB_PASSWORD` is validated as a required env var (`production.py:25`).

### 1008 - Database Port Exposed to Internet
**Severity:** CRITICAL | **Result: PASS**

- RDS is within AWS VPC private subnets.
- Security groups restrict access to application servers (ECS tasks) only.
- Database port 5432 is not exposed to the internet.

### 1009 - Database Audit Logging Disabled
**Severity:** MEDIUM | **Result: FAIL**

- No configuration found for PostgreSQL audit logging (pgaudit extension) on RDS.
- The application logs security events to a `security` logger (`core/exceptions.py`, `core/middleware.py`), but database-level audit logging (DDL, DCL, connection events) is not explicitly configured.
- No evidence of RDS Enhanced Monitoring or pgaudit parameter group configuration in the codebase.

### 1010 - Database Query Timeout Not Set
**Severity:** MEDIUM | **Result: PASS**

- Production settings configure `statement_timeout=30000` (30 seconds) via PostgreSQL connection options (`production.py:192`).
- Connection timeout is set to 10 seconds (`connect_timeout: 10`, `production.py:191`).
- Celery tasks have a global time limit of 30 minutes (`CELERY_TASK_TIME_LIMIT = 30 * 60`, `base.py:406`).

### 1011 - Database Connection Pool Exhaustion
**Severity:** HIGH | **Result: PASS**

- PgBouncer sidecar is deployed with each ECS task in `transaction` pool mode, which efficiently manages connection pooling.
- `CONN_MAX_AGE = 600` (10 min) configured in production (`production.py:189`).
- Redis cache connection pool has `max_connections: 50` with `retry_on_timeout: True` and 5-second socket timeouts (`base.py:415-422`).
- Previous DB connection exhaustion during deploy rollover was permanently fixed with PgBouncer (per memory notes).

### 1013 - Database Trigger Privilege Escalation

### 1014 - Database View Leaking Sensitive Data

### 1015 - Database Row-Level Security Missing

### 1039 - Database Transaction Isolation Level
**Severity:** MEDIUM | **Result: PASS**

- PostgreSQL default isolation level is READ COMMITTED, which is appropriate for most operations.
- Critical sections use `select_for_update()` for row-level locking:
  - Store purchases (`apps/store/services.py:159, 385-389`)
  - League standings (`apps/leagues/services.py:119, 534-546`)
  - Social event capacity (`apps/social/views.py:2584`)

### 1040 - Database Deadlock Exploitation
**Severity:** MEDIUM | **Result: FAIL**

- No explicit deadlock retry logic found in the codebase.
- Multiple views use `select_for_update()` but without consistent lock ordering documentation.
- No database deadlock monitoring or alerting configuration found.
- PostgreSQL handles deadlocks by aborting one transaction, but the application does not retry.

## Advanced API Security (1041-1100)

### 1041 - API Key Rotation Not Enforced
**Severity:** MEDIUM | **Result: FAIL**

- Third-party API keys (OpenAI, Stripe, Agora, VAPID) are stored in AWS Secrets Manager.
- No automated rotation policy or expiration tracking for these keys.
- `OPENAI_API_KEY`, `STRIPE_SECRET_KEY`, `AGORA_APP_CERTIFICATE` have no defined rotation schedule.
- JWT tokens have proper expiration (15 min access, 7 day refresh) with rotation and blacklisting (`base.py:512-519`).

### 1042 - API Key Scope Too Broad
**Severity:** HIGH | **Result: N/A**

- The application does not expose API keys to end users. Authentication is JWT-based.
- Third-party API keys (OpenAI, Stripe) are server-side only and not exposed to clients.

### 1043 - API Key Shared Across Environments
**Severity:** HIGH | **Result: FAIL**

- Production secrets are in AWS Secrets Manager (`stepora/backend-env`).
- VPS (preprod) uses its own `.env` file.
- **However**, `stepora-mobile/src/config.js:9` hardcodes `API_BASE: 'https://api.stepora.app'` (production URL) directly in the mobile source code. If the same build is used for dev testing, it could hit production.
- No evidence of separate Stripe test vs live keys per environment in the code (though Stripe keys are per-environment by nature).

### 1044 - API Response Time Side Channel
**Severity:** LOW | **Result: FAIL**

- Login endpoint performs user lookup first, then password check. Non-existent users return faster than existing users with wrong passwords.
- No constant-time comparison or artificial delays found.
- Rate limiting (5/minute on auth endpoints) provides partial mitigation.

### 1045 - API Enumeration via Sequential IDs
**Severity:** MEDIUM | **Result: PASS**

- All models use UUIDv4 primary keys: `id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False)` found across gamification, store, AI, dreams, and other models.
- `DEFAULT_AUTO_FIELD = "django.db.models.BigAutoField"` is set but overridden by explicit UUID fields on all user-facing models.
- API responses expose UUIDs, not sequential integers.

### 1046 - API Batch Request Abuse
**Severity:** MEDIUM | **Result: PASS**

- Batch sync endpoint exists (`core/sync_views.py`) but is rate-limited (`sync: 10/minute`).
- No evidence of unlimited batch operations. The batch sync view processes an offline queue replay.
- Mark-all-as-read for notifications (`apps/notifications/views.py`) is user-scoped.

### 1047 - API Partial Response Data Leak
**Severity:** MEDIUM | **Result: PASS**

- No field selection parameters (`?fields=`) are supported. DRF serializers define explicit field lists.
- Default renderer is JSON only (`rest_framework.renderers.JSONRenderer`).
- Sensitive fields (password hashes, encryption keys) are excluded from serializers via `write_only=True` or explicit `fields` lists.

### 1048 - API Sorting/Filtering Injection
**Severity:** MEDIUM | **Result: PASS**

- `ordering_fields` are explicitly defined as allowlists on viewsets (e.g., `["price", "created_at", "name", "rarity"]` in `store/views.py:140`).
- `filterset_fields` are explicitly defined (e.g., `["category__slug", "item_type", "rarity"]` in `store/views.py:138`).
- DRF's `OrderingFilter` and `DjangoFilterBackend` only accept fields listed in these allowlists.

### 1049 - API Search Parameter Injection
**Severity:** MEDIUM | **Result: PASS**

- Search input is passed through elasticsearch_dsl query builders or Django ORM `icontains`, both of which parameterize input.
- Search queries are user-scoped (`s.filter("term", user_id=str(user.id))`) in `apps/search/services.py`.
- Minimum query length enforced (2 characters) in `GlobalSearchView`.
- Search is rate-limited (`search: 15/minute`).

### 1050 - API Nested Resource Depth Abuse
**Severity:** MEDIUM | **Result: PASS**

- DRF serializer depth is controlled explicitly. No `?include=` or `?expand=` parameters are supported.
- Nested data is fetched via `select_related` / `prefetch_related` with explicit field sets.
- No recursive or deeply nested serializers found.

### 1051 - API Field Selection Injection
**Severity:** MEDIUM | **Result: N/A**

### 1052 - API Aggregation Pipeline Injection
**Severity:** HIGH | **Result: N/A**

### 1053 - API File Upload Multipart Parsing
**Severity:** HIGH | **Result: PASS**

- File uploads validate:
  - Content-type against allowlists (`apps/users/views.py:302-303`, `apps/dreams/views.py:1633-1635`)
  - File size limits (5MB for avatars, 110MB global per `DATA_UPLOAD_MAX_MEMORY_SIZE`)
  - Magic bytes validation to prevent content-type spoofing (`apps/users/views.py:316-331`, `apps/dreams/views.py:1649`)
  - Filenames sanitized with UUID replacement (`apps/users/views.py:339-342`)

## Critical Findings Summary

### FAIL Items Requiring Attention (sorted by severity):

1. **1015 - Row-Level Security Missing (HIGH):** No PostgreSQL RLS policies. Data isolation depends entirely on application-level ORM filtering. Consider adding RLS for defense-in-depth on sensitive tables (dreams, AI conversations, user data).

2. **1019 - Redis Without Password (CRITICAL concern, depends on infra):** Verify that AWS ElastiCache has AUTH enabled and in-transit encryption configured. Network isolation via VPC provides mitigation but AUTH adds defense-in-depth.

3. **1060 - WebSocket Connection Limits (MEDIUM):** No per-user concurrent WebSocket connection limits. An attacker could open many connections to exhaust server resources.

4. **1040 - Deadlock Exploitation (MEDIUM):** No deadlock retry logic. Multiple `select_for_update()` usages without documented lock ordering could be exploited.

5. **1082 - Bulk Delete Without Confirmation (HIGH):** Account deletion and bulk notification operations lack re-authentication or confirmation.

6. **1009 - Database Audit Logging (MEDIUM):** No RDS pgaudit configuration found. Important for compliance and incident forensics.

7. **1006 - Hardcoded Credentials (HIGH):** Test file contains hardcoded dev DB password; Dockerfile contains placeholder encryption key in image layer.

8. **1031 - Migration Rollback Risk (MEDIUM):** No tested rollback procedures for migrations.

9. **1033 - Timing Side Channel (MEDIUM):** User enumeration possible via login response timing differences.

10. **1041 - API Key Rotation (MEDIUM):** No rotation schedule for third-party API keys (OpenAI, Stripe, Agora).

11. **1043 - API Key Shared Across Environments (HIGH):** Mobile app hardcodes production API URL.

12. **1044 - Response Time Side Channel (LOW):** Login timing reveals user existence.

13. **1056 - API Version Downgrade (HIGH):** Both `/api/v1/` and `/api/` serve same code (low risk currently, but no sunset plan).

14. **1085 - Healthcheck Info Disclosure (LOW):** Public health check reveals DB/cache status and latency.

15. **1038 - Bulk Operation Without Limit (MEDIUM):** Some bulk deletes lack count validation.
