## Network Security (601-630)

### VULN-601: Man-in-the-Middle (MITM) Attack
**Severity:** CRITICAL | **Status: PASS**

- HSTS header set with `max-age=31536000; includeSubDomains; preload` in `SecurityHeadersMiddleware` (`core/middleware.py:188-190`).
- Production settings enforce `SESSION_COOKIE_SECURE = True`, `CSRF_COOKIE_SECURE = True` (`config/settings/production.py:47-48`).
- `SECURE_PROXY_SSL_HEADER` configured for ALB TLS termination (`production.py:45`).
- JWT refresh cookie set with `Secure=True` in production (`production.py:62`).
- Frontend auth uses `crypto.getRandomValues()` for OAuth state parameters (not `Math.random()`).
- **Gap:** No certificate pinning on mobile/Capacitor apps. This is noted but not a FAIL since ALB/CloudFront handle TLS.

### VULN-602: ARP Spoofing
**Severity:** HIGH | **Status: N/A**

Network switch-level control. Application runs on AWS ECS Fargate; ARP is managed by AWS infrastructure.

### VULN-603: DNS Spoofing/Cache Poisoning
**Severity:** CRITICAL | **Status: N/A**

DNS is managed by Cloudflare. DNSSEC and DNS resolver configuration are infrastructure-level concerns outside application code.

### VULN-604: BGP Hijacking
**Severity:** CRITICAL | **Status: N/A**

BGP routing is managed by AWS and Cloudflare at the infrastructure level.

### VULN-605: SSL Stripping

### VULN-680: Unicode Normalization Attack
**Severity:** MEDIUM | **Status: FAIL**

- Display name patterns allow Unicode ranges (`\u00C0-\u024F`, `\u1E00-\u1EFF`) without normalizing input first.
- Two visually identical display names using different Unicode representations could bypass the uniqueness check.
- No consistent Unicode normalization at the input boundary.

### Additional Observations

1. **No certificate pinning** on Capacitor/mobile apps (VULN-601 gap). While not a FAIL since TLS is enforced server-side, certificate pinning would add defense-in-depth.

2. **`Math.random()` for optimistic message IDs** (`api.js:519` in both frontend and mobile): Used for temporary client-side IDs replaced by server-generated UUIDs. Not a security issue but worth noting.

3. **Fixed salt for backup code hashing** (`two_factor.py:39`): Uses `hashlib.sha256(b"stepora-backup-codes").digest()` as a fixed salt. A per-user salt would be stronger but current approach with 100,000 PBKDF2 iterations is adequate.
