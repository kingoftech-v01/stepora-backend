## Defensive Security (1201-1240)

### 1201 - Input Validation Not Centralized
**Severity:** HIGH | **Result: PASS**

- Backend uses a centralized `core/validators.py` module with `validate_uuid()`, `validate_pagination_params()`, `validate_search_query()`, `validate_display_name()`, `validate_location()`, `validate_text_length()`, `validate_url_no_ssrf()`.
- Centralized sanitization in `core/sanitizers.py` using `nh3` (Rust-based HTML sanitizer).
- DRF serializers enforce validation at the API layer with consistent patterns.
- Server-side validation exists independently of client-side validation in both frontend and mobile.

---

### 1202 - Output Encoding Not Consistent
**Severity:** HIGH | **Result: FAIL**

- Django templates use auto-escaping by default (PASS for backend).
- Frontend: 3 files render AI-generated content as raw HTML in AI chat screens (`AIChatMobile.jsx`, `AIChatTablet.jsx`, `AIChatDesktop.jsx`) using unsafe HTML rendering.
- Frontend: `RichTextEditor.jsx` uses direct DOM innerHTML assignment.
- Mobile (React Native): No unsafe HTML rendering usage found (PASS).
- **Issue:** AI chat output rendered as raw HTML should be sanitized client-side (e.g., DOMPurify) before rendering.

---

### 1203 - Error Handling Reveals Internal State
**Severity:** MEDIUM | **Result: PASS**

- Custom exception handler in `core/exceptions.py` hides internal error details for 500 errors in production: masks internal message with generic "An internal error occurred."
- `DEBUG = False` enforced in `config/settings/production.py` with startup validation.
- Custom error pages configured. No stack traces sent to clients.
- Sentry integration configured with `send_default_pii=False`.

---

### 1204 - Fail-Open vs Fail-Close Misconfiguration
**Severity:** CRITICAL | **Result: PASS**

- DRF default permission is `IsAuthenticated` -- fails closed when auth is unavailable.
- Rate limiter (Redis-backed) -- if Redis fails, DRF's built-in throttling falls back to denying (no fail-open).
- Auth middleware (`EmailVerificationMiddleware`) catches JWT exceptions and passes through to DRF which returns 401 (fail-closed).
- Account lockout via cache: if cache is unavailable, `cache.get()` returns None (no lockout record), but login still requires valid credentials (fail-closed for auth, fail-open for lockout only).

---

### 1205 - Defense in Depth Missing
**Severity:** HIGH | **Result: PASS**

- Multiple security layers observed:
  - Network: ALB, CORS, Origin validation middleware, Admin IP restriction middleware.
  - Application: JWT auth, permission classes, rate limiting, input validation, output sanitization.
  - Data: Field-level encryption (`encrypted_model_fields`), DB SSL (`sslmode=require`), S3 for media.
  - Transport: HSTS, Secure cookies, SameSite cookies.
  - CSP, X-Frame-Options, X-Content-Type-Options headers via `SecurityHeadersMiddleware`.

---

### 1206 - Least Privilege Not Applied
**Severity:** HIGH | **Result: FAIL**

- Docker container runs as non-root user `appuser` (PASS).
- However, DB credentials are shared across all ECS services (backend, celery, celery-beat) -- no separate read-only DB user for workers.
- `ALLOWED_HOSTS=*` in production ECS (documented workaround for ALB health checks, but overly permissive).
- AWS IAM roles not audited in code, but ECS task roles exist.
- **Issue:** Single DB user with full privileges across all services. Workers should use restricted DB accounts.

### 1207 - Separation of Duties Missing
**Severity:** MEDIUM | **Result: FAIL**

- Single developer appears to have full access to deploy, merge, and manage infrastructure.
- No branch protection rules enforced in CI/CD (deploy workflow triggers on push to main/development without required reviews).
- `deploy-backend.yml` runs on `self-hosted` runners -- single point of control.
- No evidence of code review requirements enforced via branch protection in the workflow files.

### 1208 - Security by Obscurity Relied Upon
**Severity:** MEDIUM | **Result: PASS**

- All endpoints have proper authentication (`IsAuthenticated` default).
- Admin panel protected by `AdminIPRestrictionMiddleware` (not just hidden URL).
- Authorization checks exist on every endpoint via DRF permission classes.
- Hidden endpoints are not used as a security mechanism.
- Obfuscated IDs (UUIDs) are backed by proper ownership checks (`IsOwner` permission).

### 1209 - Allowlist vs Denylist (Wrong Approach)
**Severity:** HIGH | **Result: PASS**

- CORS uses allowlist (`CORS_ALLOWED_ORIGINS` with specific origins).
- Input validation uses allowlist regex patterns (e.g., `DISPLAY_NAME_PATTERN`, `COUPON_CODE_PATTERN`).
- URL validation uses allowlisted schemes (`http`, `https`).
- `sanitize_url()` in `core/sanitizers.py` uses a denylist approach for dangerous patterns -- however this is a secondary defense behind the allowlist-based scheme validation.
- File upload validation would need further review, but DRF serializers define allowed types.

### 1210 - Trusted Boundary Not Defined
**Severity:** HIGH | **Result: FAIL**

- No formal trust boundary documentation found.
- Third-party API responses (OpenAI, Stripe, Google) are used with varying levels of validation.
- Inter-service communication (backend to Redis, backend to RDS) uses trusted connections without mutual TLS within VPC.
- Stripe webhook signature verification is present (PASS for that specific case).
- **Issue:** No documented trust boundaries. OpenAI API responses are processed with some validation (`core/ai_validators.py`) but this is inconsistent.

### 1211 - Anti-Pattern: Catching All Exceptions Silently
**Severity:** HIGH | **Result: FAIL**

- 51 occurrences of broad exception handling (`except Exception`) across 20 Python files.
- Several bare `except` blocks exist, particularly in:
  - `core/permissions.py` (lines 73-76, 82-83) -- silently catches exceptions during shared dream/collaborator checks.
  - `core/middleware.py` (LastActivityMiddleware, line 322) -- catches exception with only debug logging.
  - `core/exceptions.py` (line 117) -- bare `except: pass` in quota reset time calculation.
  - `integrations/openai_service.py` -- 9 occurrences of broad exception handling.
- **Issue:** Several security-relevant exception handlers silently swallow errors, particularly in permission checks and AI integration code.

### 1212 - Anti-Pattern: Logging Sensitive Data
**Severity:** HIGH | **Result: PASS**

- Logging statements reviewed: no passwords, tokens, or secrets are logged.

- FCM token logging truncates to first 20 chars.
- Sentry configured with `send_default_pii=False`.

- Auth views log IP and path but not credentials.

### 1213 - Anti-Pattern: Hardcoded Timeout Values
**Severity:** MEDIUM | **Result: FAIL**

- JWT access token lifetime hardcoded to 15 minutes in `base.py`.
- JWT refresh token lifetime hardcoded to 7 days.
- Session timeout hardcoded to 24 hours (`SESSION_COOKIE_AGE = 86400`).
- Email verification key max age hardcoded to 3 days.
- Password reset max age hardcoded to 1 hour.
- Rate limit values hardcoded in settings (not in env vars).
- `TOKEN_EXPIRY_HOURS` is configurable via env var (partial mitigation).
- **Issue:** Critical security timeouts (JWT, session) are not configurable via environment variables, requiring code deployment to change.

**Severity:** MEDIUM | **Result: PASS**

## Recommendations (Top 10, prioritized)

1. **Add SAST/dependency scanning to CI/CD** (1234, 1250, 1295) -- Add Bandit, Safety/pip-audit, and Dependabot to the pipeline.
2. **Deploy a WAF** (1274) -- Enable AWS WAF on the ALB with OWASP Top 10 managed rules.
3. **Sanitize AI chat output client-side** (1202) -- Add DOMPurify before unsafe HTML rendering in AI chat screens.
4. **Create an Incident Response Plan** (1244, 1245, 1246, 1247) -- Document IR procedures, roles, and communication plans.
5. **Add monitoring/alerting** (1222, 1223, 1267) -- Configure CloudWatch alarms for auth failures, error rate spikes, and API anomalies.
6. **Fix broad exception handling** (1211) -- Replace bare `except` blocks with specific exception types; add logging.
7. **Make security timeouts configurable** (1213) -- Move JWT lifetimes and session timeout to environment variables.
8. **Add CAPTCHA** (1225) -- Integrate reCAPTCHA v3 or Turnstile on registration and login.
9. **Implement password history** (1229) -- Add a `PasswordHistory` model to prevent reuse.
10. **Add session management for users** (1230) -- Allow users to view and terminate active sessions.
