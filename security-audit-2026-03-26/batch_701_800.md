# Security Audit Report: Vulnerabilities 701-800

**Date:** 2026-03-26
**Repos Audited:**
- `/root/stepora` (Django backend)
- `/root/stepora-frontend` (React/Vite frontend)
- `/root/stepora-mobile` (React Native mobile)

**Methodology:** Each vulnerability checklist item inspected against codebase. Verdicts: PASS, FAIL, N/A.

---

## Category: Memory Safety (701-720)

### Vuln 701 - Buffer Overflow
**Verdict: N/A**
All three repos are written in Python (backend), JavaScript/React (frontend), and JavaScript/React Native (mobile). These are memory-managed languages with no manual buffer management. No C/C++ extensions found in any repo.

### Vuln 702 - Stack Overflow
**Verdict: N/A**
Python, JavaScript, and React Native handle stack management automatically. No unbounded recursion patterns found. Python has a default recursion limit (1000). Not applicable to this technology stack.

### Vuln 703 - Heap Overflow
**Verdict: N/A**
No C/C++ or manual memory management code present. Python and JavaScript use garbage collection with managed heaps.

### Vuln 704 - Use-After-Free
**Verdict: N/A**
Not applicable to Python/JavaScript. Memory is garbage-collected.

### Vuln 705 - Double Free
**Verdict: N/A**
Not applicable. No manual memory deallocation in any repo.

### Vuln 706 - Null Pointer Dereference
**Verdict: N/A**
Python raises `AttributeError`/`TypeError` on None access (not exploitable for code execution). JavaScript has similar safe behavior. Not a memory safety concern in these languages.

### Vuln 707 - Dangling Pointer
**Verdict: N/A**
No manual pointer management. Garbage-collected languages only.

### Vuln 708 - Format String Vulnerability
**Verdict: N/A**
Python uses `str.format()` and f-strings which are not vulnerable to format string attacks in the C sense. No `printf`-family calls.

### Vuln 709 - Integer Overflow Leading to Buffer Overflow
**Verdict: N/A**
Python integers have arbitrary precision (no overflow). JavaScript numbers are IEEE 754 doubles. No allocation size calculations that could overflow.

### Vuln 710 - Off-By-One Error
**Verdict: N/A**
While off-by-one logic bugs can exist in any language, they do not lead to memory corruption in Python/JavaScript. Not a memory safety issue here.

### Vuln 711 - Uninitialized Memory Read
**Verdict: N/A**
Python and JavaScript initialize variables. No uninitialized memory access possible in these languages.

### Vuln 712 - Memory Leak (Resource Exhaustion)
**Verdict: PASS**
- Backend uses Django with connection pooling (`CONN_MAX_AGE=600`), Celery `max-tasks-per-child=1000` (worker recycling), and Redis cache with `TIMEOUT=300`.
- Docker containers have health checks enabling restart on failure.
- ECS Fargate manages container lifecycle with health check grace periods.
- No circular reference patterns identified in application code.

### Vuln 713 - Stack Canary Bypass
**Verdict: N/A**
No compiled binaries in the application code. Container base images (python:3.11-slim, node:20-alpine) inherit OS-level protections.

### Vuln 714 - ASLR Bypass
**Verdict: N/A**
Application runs in managed runtime environments (CPython, Node.js). ASLR is an OS-level concern managed by container host and AWS Fargate.

### Vuln 715 - DEP/NX Bypass
**Verdict: N/A**
No native code execution. DEP/NX is managed by the underlying OS and Fargate infrastructure.

### Vuln 716 - Return-Oriented Programming (ROP)
**Verdict: N/A**
No compiled native binaries. Not applicable to interpreted language applications.

### Vuln 717 - Type Confusion
**Verdict: N/A**
Python and JavaScript are dynamically typed but do not suffer from memory-layout type confusion. Django ORM and DRF serializers provide type validation.

### Vuln 718 - Spectre/Meltdown Side Channel
**Verdict: N/A**
Infrastructure-level concern. AWS Fargate manages host-level CPU mitigations. Application code cannot mitigate or detect this.

### Vuln 719 - Rowhammer Attack
**Verdict: N/A**
Hardware-level attack. AWS manages physical infrastructure. RDS uses ECC memory. Not applicable to application code.

### Vuln 720 - Memory-Safe Language Misuse (unsafe blocks)
**Verdict: N/A**
No Rust, Go, or C# code found. No `unsafe` blocks, CGo, or FFI calls. All code is pure Python and JavaScript.

## Category: Container + Orchestration (721-750)

### Vuln 721 - Container Image Not Signed
**Verdict: FAIL**
- No Docker Content Trust, cosign, or Notary signing found in CI/CD pipeline (`deploy-backend.yml`).
- Images are pushed to ECR without cryptographic signatures.
- No admission controller or signature verification at deployment time.

### Vuln 722 - Container Registry Without Auth
**Verdict: PASS**
- ECR (Amazon Elastic Container Registry) requires IAM authentication.
- CI/CD uses `aws ecr get-login-password` for authenticated push.
- ECR is private and not publicly accessible.
- TLS enforced by default on ECR.

### Vuln 723 - Privileged Container
**Verdict: PASS**
- No `privileged: true` in `docker-compose.yml` or `docker-compose.prod.yml`.
- Dockerfile creates and uses non-root user `appuser` (UID 1000).
- ECS Fargate does not support privileged mode by design.
- Nginx container uses `read_only: true`.

### Vuln 724 - Container Host Filesystem Mount
**Verdict: FAIL**
- Development `docker-compose.yml` mounts `.:/app` (entire project directory) into web, celery, celery-beat, and daphne containers (lines 82, 142, 195, 238).
- This exposes source code, `.env` file, and `firebase-service-account.json` to the container.
- Production ECS uses container images without host mounts (PASS for production).
- No `docker.sock` mount found (good).

### Vuln 725 - Container Network Not Isolated
**Verdict: FAIL**
- All services in `docker-compose.yml` share a single flat network (`stepora_network`).
- No network policies or segmentation between database, Redis, Elasticsearch, and application containers.
- Production uses ECS Fargate in public subnets with security groups providing some isolation, but within the same VPC network.

### Vuln 733 - Container Runtime Vulnerability

### Vuln 737 - Kubernetes Secret Not Encrypted at Rest
**Verdict: N/A**
No Kubernetes Secrets. AWS Secrets Manager is used (encrypted at rest by default with KMS).

### Vuln 738 - Container Log Injection
**Verdict: FAIL**
- Backend logging uses Python's `logging` module with string formatting, not structured JSON logging in all places.
- Production logging config (`production.py`) uses `{levelname} {asctime} {module} {message}` format (plain text, not JSON).
- User input logged in some views (e.g., error messages, request data) without explicit sanitization for newline injection.
- No log integrity verification.

### Vuln 739 - Container Resource Limits Not Set
**Verdict: FAIL**
- In `docker-compose.yml`, `db`, `redis`, `elasticsearch`, and `nginx` have resource limits.
- **`web`, `celery`, `celery-beat`, and `daphne` containers have NO resource limits** in `docker-compose.yml`.
- Production ECS Fargate tasks have CPU/memory limits defined in task definitions (managed outside the repo).

### Vuln 740 - Docker Compose Secrets in YAML
**Verdict: PASS**
- `docker-compose.yml` uses `${DB_PASSWORD}`, `${REDIS_PASSWORD}`, `${DJANGO_SECRET_KEY}` and other environment variable references, not hardcoded values.
- `.env` is in `.gitignore`.
- Production uses AWS Secrets Manager.

### Vuln 741 - Docker Build Cache Poisoning
**Verdict: PASS**
- Builds use GitHub Actions on self-hosted runner with ECR.
- No shared remote build cache exposed externally.
- Multi-stage build isolates build dependencies from runtime.

### Vuln 742 - Multi-Stage Build Secret Leakage
**Verdict: FAIL**
- `Dockerfile` line 60: `ARG FIELD_ENCRYPTION_KEY="oWaFzG4bta5qhklgSfvMxFPqwqvy__35IrKHEPURBkg="` exposes a default/placeholder encryption key as a build argument.
- Build args are visible in `docker history`. While this is a placeholder overridden at runtime, it creates a pattern where real keys could leak.
- `deploy-backend.yml` line 77 passes `FIELD_ENCRYPTION_KEY` as `--build-arg`, which persists in image metadata.
- `.dockerignore` should be reviewed to ensure `.env` and credential files are excluded.

### Vuln 743 - Container Health Check Missing
**Verdict: PASS**
- All containers in `docker-compose.yml` have health checks defined.
- `healthcheck.sh` implements role-aware checks (HTTP for web, process check for worker/beat).
- Dockerfile includes `HEALTHCHECK` instruction.
- ECS tasks have health check configuration with grace periods.

### Vuln 744 - Container Graceful Shutdown Missing
**Verdict: PASS**
- `entrypoint.sh` uses `exec` for all process launches (daphne, celery worker, celery beat), ensuring PID 1 receives SIGTERM directly.
- Daphne and Celery both handle SIGTERM gracefully by default.
- No explicit `trap` for SIGTERM, but `exec` replaces the shell, so signals reach the application process directly.

### Vuln 745 - Kubernetes CronJob Privilege
**Verdict: N/A**
No Kubernetes CronJobs. Celery Beat with DatabaseScheduler is used for scheduled tasks.

### Vuln 746 - Kubernetes Role Binding Escalation
**Verdict: N/A**
No Kubernetes RBAC.

### Vuln 747 - Container Image Layer Cache Attack
**Verdict: PASS**
- Base images are from official Docker Hub sources (python, node, nginx).
- ECR private registry for application images.
- CI/CD uses commit SHA as image tag for traceability.

### Vuln 748 - Container Startup Probe Missing
**Verdict: N/A**
No Kubernetes startup probes (ECS concept is health check grace period). ECS health check has `start_period` of 30-40 seconds configured.

### Vuln 749 - Kubernetes Namespace Isolation Missing
**Verdict: N/A**
No Kubernetes namespaces.

### Vuln 750 - Container Drift Detection Missing
**Verdict: FAIL**
- No runtime security monitoring (Falco, Sysdig, or equivalent).
- Nginx container has `read_only: true` (good), but web, celery, daphne containers do not.
- No file integrity monitoring in production.
- ECS exec is available (requires `ssmmessages:*` IAM permissions) but no auditing of exec sessions beyond CloudTrail.

## Critical Findings

### FAIL Items Requiring Attention

1. **Vuln 721 - Container Image Not Signed:** No image signing in CI/CD. Add cosign or Docker Content Trust.

2. **Vuln 724 - Host Filesystem Mount (dev):** Development docker-compose mounts `.:/app` exposing `.env` and `firebase-service-account.json` to containers. Production is not affected (ECS uses container images).

3. **Vuln 725 - Container Network Not Isolated:** All dev services share a flat network. Consider network segmentation for database and Redis.

4. **Vuln 738 - Container Log Injection:** Production uses plain text logging (not structured JSON). User input in logs is not sanitized for newline injection.

5. **Vuln 739 - Resource Limits Missing:** Web, celery, celery-beat, and daphne containers lack resource limits in docker-compose.yml (db, redis, elasticsearch, nginx do have them).

6. **Vuln 742 - Multi-Stage Build Secret Leakage:** `FIELD_ENCRYPTION_KEY` passed as `--build-arg` and exposed in `docker history`. Use BuildKit `--mount=type=secret` instead.

7. **Vuln 750 - No Drift Detection:** No runtime security monitoring (Falco/Sysdig). Application containers are not read-only (only nginx is).

8. **Vuln 780 - No Security Testing in CI:** No SAST (Bandit, Semgrep), DAST, or dependency scanning in CI/CD. No WAF deployed.

9. **Vuln 782 - Insufficient Log Retention:** Production logs retained only 30 days in CloudWatch. PCI-DSS requires 12 months. No SIEM integration.

10. **Vuln 783 - No Security Testing Program:** No penetration testing, vulnerability scanning, or IDS evidence.

11. **Vuln 784/792/793 - Missing Governance Docs:** No ISMS, risk assessment, incident response plan, or formal security policies.

12. **Vuln 797/798/799 - Detect/Respond/Recover Gaps:** No SIEM, no incident response plan, no documented recovery procedures with RTO/RPO.

### Positive Findings

- Non-root container user (`appuser`, UID 1000) in Dockerfile.
- ECR requires IAM authentication.
- No privileged containers.
- Health checks on all containers.
- Graceful shutdown via `exec` in entrypoint.
- Secrets managed via environment variables (not hardcoded in compose YAML).
- AWS Secrets Manager for production secrets.
- Strong authentication (JWT, rate limiting, token blacklisting).
- Field-level encryption for PII.
- TLS enforced (HSTS, secure cookies, DB SSL).
- No cardholder data stored (Stripe handles payments).
- Sentry configured with `send_default_pii=False`.
- Account deletion and data export functionality implemented.
