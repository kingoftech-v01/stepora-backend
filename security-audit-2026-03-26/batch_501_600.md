## Microservices (501-530)

### V-501: Service-to-Service Auth Missing
**Severity:** CRITICAL | **Category:** Microservices | **Result: N/A**

Stepora is a monolithic Django application, not a microservices architecture. All backend services (web, celery worker, celery beat) share the same codebase and database. There is no inter-service API communication to authenticate. The only "services" are ECS tasks running different roles of the same Django app.

---

### V-502: Internal API Without Authentication
**Severity:** CRITICAL | **Category:** Microservices | **Result: N/A**

Not applicable -- monolithic architecture. There are no internal APIs exposed between services. The single Django app handles all API requests behind ALB. The health check endpoints (`/health/`, `/health/liveness/`, `/health/readiness/`) are intentionally unauthenticated but only return status data.

---

### V-503: Service Mesh Misconfiguration
**Severity:** HIGH | **Category:** Microservices | **Result: N/A**

No service mesh (Istio, Linkerd, Consul Connect) is deployed. ECS Fargate tasks communicate only via shared PostgreSQL (through PgBouncer sidecar) and Redis -- not through a service mesh.

---

### V-504: Sidecar Proxy Bypass
**Severity:** HIGH | **Category:** Microservices | **Result: N/A**

No sidecar proxy mesh is in use. The only sidecar is PgBouncer for database connection pooling, which is not a security proxy. Traffic flows: Internet -> ALB -> ECS task (port 8000).

---

### V-505: Service Discovery Poisoning
**Severity:** HIGH | **Category:** Microservices | **Result: N/A**

No service discovery system (Consul, Eureka) is deployed. ECS services are registered via ALB target groups managed by AWS, which is not susceptible to external registration poisoning.

---

### V-506: Circuit Breaker Not Implemented
**Severity:** MEDIUM | **Category:** Microservices | **Result: FAIL**

No circuit breaker pattern is implemented for external service calls (OpenAI API, Stripe API, Google Calendar API). The code uses retries with `max_retries` and `countdown` delays on Celery tasks (`apps/ai/tasks.py`, `apps/notifications/tasks.py`, `core/tasks.py`), and `tenacity.retry` for OpenAI calls (`integrations/openai_service.py`), but there is no circuit breaker to stop calling a persistently failing service. A prolonged OpenAI outage would exhaust Celery workers with retry storms.

**Evidence:** `integrations/openai_service.py` line 44 uses `tenacity.retry` with wait/stop but no circuit breaker. Multiple tasks have `max_retries=3` with fixed countdown delays.

---

### V-507: Retry Storm (Cascading Retries)
**Severity:** MEDIUM | **Category:** Microservices | **Result: FAIL**

Celery tasks use retries with fixed countdowns (30s, 60s, 300s) but without exponential backoff with jitter in most cases. Multiple tasks at different layers can retry simultaneously. The `tenacity` retry on `openai_service.py` does use exponential backoff, but Celery task retries on top of that create layered retries.

**Evidence:** `apps/ai/tasks.py` lines 141-147: three separate `self.retry(countdown=30/30/60)` without jitter. `apps/notifications/tasks.py` line 190: `self.retry(countdown=60)` fixed delay.

---

### V-508: Distributed Tracing Data Leak
**Severity:** MEDIUM | **Category:** Microservices | **Result: N/A**

No distributed tracing system (Jaeger, Zipkin, Datadog APM) is deployed. Sentry is configured for error tracking with `send_default_pii=False` in production settings, which is correct.

### V-509: Config Server Exposed
**Severity:** CRITICAL | **Category:** Microservices | **Result: N/A**

No centralized config server (Spring Cloud Config, Consul KV, etcd) is deployed. Configuration is managed through environment variables injected via AWS Secrets Manager into ECS task definitions.

### V-510: Secret Rotation Not Automated
**Severity:** HIGH | **Category:** Microservices | **Result: FAIL**

Secrets in AWS Secrets Manager (`stepora/backend-env`) do not have automatic rotation configured. Database passwords, API keys (OpenAI, Stripe, Agora), and encryption keys are static. No rotation Lambda functions are deployed.

**Evidence:** Memory notes confirm secrets are in Secrets Manager but no rotation configuration is mentioned. `FIELD_ENCRYPTION_KEY` is hardcoded as a build-time ARG default in `Dockerfile` line 60.

### V-511: Health Check Endpoint Leaks Info
**Severity:** MEDIUM | **Category:** Microservices | **Result: PASS**

Health check endpoints are properly implemented:

- `/health/readiness/` returns `{"status": "ready"}` or `{"status": "not ready", "reason": "database unavailable"}`.
- `/health/` returns service status (up/down) for database and cache but no connection strings, versions, or hostnames.

**Evidence:** `core/views.py` lines 16-51.

### V-512: Canary/Blue-Green Deployment Rollback

Previous task definition revisions are retained by ECS but rollback is manual.

**Evidence:** `.github/workflows/deploy-backend.yml` lines 98-150. No `deploymentConfiguration` with `alarms` or `circuitBreaker` in the deployment step.

### V-516: Saga Pattern Compensation Failure

### V-533: S3 Object ACL Public Read
**Severity:** HIGH | **Category:** Cloud | **Result: PASS**

**Evidence:** `config/settings/production.py` line 93: `AWS_QUERYSTRING_AUTH = False`. This means media files are accessible via direct URL if the bucket allows it.

**Note:** While ACLs are disabled, `AWS_QUERYSTRING_AUTH = False` combined with no pre-signed URLs means media access depends entirely on bucket policy.
