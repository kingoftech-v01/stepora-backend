### 1162 - TOTP Window Too Wide
**Severity:** MEDIUM | **Status: PASS**

- `totp.verify(code, valid_window=1)` in both locations
- 30-second default time step (pyotp default per RFC 6238)

- Encryption key stored in AWS Secrets Manager, not in code
- Backup codes are hashed with PBKDF2 (100,000 iterations) before storage

### 1169 - Biometric Bypass via Replay
**Severity:** HIGH | **Status: PASS**

The `createSignature` function creates a cryptographic challenge-response that cannot be replayed. `allowDeviceCredentials: true` is set, which allows PIN fallback (see 1170).

### 1170 - Biometric Fallback to PIN (weak)
**Severity:** MEDIUM | **Status: FAIL**

`allowDeviceCredentials: true` is configured in the biometrics service, allowing fallback to device PIN/password when biometric fails. This weakens security since a 4-6 digit PIN is easier to brute-force than biometrics. No additional rate limiting or lockout is implemented at the app level for biometric fallback.

- `allowDeviceCredentials: true` in `biometrics.js` line 26
- OS-level lockout after failed PIN attempts provides some protection
- No app-level rate limiting on biometric fallback

### 1196 - Federated Identity Provider Compromise
**Severity:** CRITICAL | **Status: FAIL**

Google and Apple are used as federated identity providers:

### 1198 - Token Sidejacking
**Severity:** HIGH | **Status: PASS**

Token transmission security is well-implemented:

- Refresh cookie has `Secure` flag in production (`_COOKIE_SECURE = True`)
- Refresh cookie is `HttpOnly` (not accessible via JavaScript)
- Access tokens in Authorization header over HTTPS only
- `SECURE_PROXY_SSL_HEADER` configured for ALB
- Tokens never in URL parameters
- `SameSite=None` for cross-origin but with `Secure` flag

### 1199 - Session Puzzling
**Severity:** HIGH | **Status: PASS**

JWT-based authentication avoids session puzzling:

- Stateless JWT tokens carry user identity, not server-side session state
- 2FA challenge tokens are signed with HMAC and bound to IP address
- Challenge tokens have 5-minute expiry and separate salt
- No cross-workflow session variable abuse possible with JWT architecture
- Redis sessions used for Django admin only (separate from API auth)

### 1200 - Authentication Logic Flaw
**Severity:** CRITICAL | **Status: PASS**

- Email verification uses `django.core.signing` with salt and max_age
- Null/empty parameter handling tested
- Default-deny: `DEFAULT_PERMISSION_CLASSES = [IsAuthenticated]`
- Rate limiting on all auth endpoints
- Login notifications sent on successful auth
- Password change triggers notification email
