## OAuth/OIDC (901-925)

### 901 - OAuth Authorization Code Interception
**Severity:** HIGH | **Result: PASS**

The app uses server-side token verification for social login (Google/Apple). Google ID tokens are verified via `google.oauth2.id_token.verify_oauth2_token()` or manual JWKS validation in `/root/stepora/core/auth/social.py`. Apple tokens are verified using Apple's JWKS endpoint. Authorization codes are not exposed client-side. Google Calendar OAuth uses PKCE (code_verifier/code_challenge) in `/root/stepora/integrations/google_calendar.py`. Redirect URIs are HTTPS in production.

- [x] Redirect URIs use HTTPS exclusively (production)
- [x] Authorization codes are handled server-side only
- [x] Client authentication is required for code exchange (Google Calendar)
- [x] PKCE is enforced for Google Calendar OAuth flow
- [x] State parameter is used in Google Calendar OAuth

---

### 902 - OAuth Implicit Flow Token Leakage
**Severity:** HIGH | **Result: FAIL**

The `PRODUCTION_CHECKLIST.md` and `TODO.md` both document that Google social login still uses `response_type=token` (implicit flow) for the frontend login flow. The frontend (`useLoginScreen.js`) uses the Google Identity Services library which returns tokens via the implicit flow. Migration to Authorization Code + PKCE is listed as a TODO but not yet implemented.

- [ ] Implicit flow is not used -- **FAIL: frontend Google login uses implicit flow**
- [ ] Migration plan is documented -- migration is documented in TODO.md

**Evidence:** `/root/stepora/TODO.md` line: "Migrate to PKCE OAuth -- Replace Google implicit flow (response_type=token) with Authorization Code + PKCE"

---

### 903 - OAuth PKCE Not Implemented
**Severity:** HIGH | **Result: FAIL**

PKCE is implemented for Google Calendar OAuth (`/root/stepora/integrations/google_calendar.py`) with proper code_verifier generation, but NOT for the primary Google social login flow used for authentication. The frontend Google/Apple social login sends ID tokens or access tokens directly to the backend without PKCE.

- [x] PKCE is implemented for Google Calendar OAuth
- [ ] PKCE is NOT enforced for primary social login flow -- **FAIL**
- [x] code_verifier has sufficient entropy (HMAC-SHA256 based, 43-128 chars)

No client secrets found in frontend or mobile source code.

- [x] Client secrets are never embedded in client-side code
- [x] Secrets are stored in secure vaults (AWS Secrets Manager) or environment variables

- [x] Mobile apps use backend proxy for OAuth token exchange

### 906 - OAuth Token Theft via Open Redirect
**Severity:** HIGH | **Result: PASS**

Only exact prefix matches are accepted. The redirect output is HTML-escaped via `html.escape()` and JSON-encoded via `json.dumps()`.

- [x] Redirect URIs are validated against allowed prefixes
- [x] Path traversal in redirect URIs is blocked (prefix matching only)
- [x] Output is properly escaped (html.escape + json.dumps)

The `TokenRefreshView` in `views.py` blacklists old refresh tokens after rotation. Refresh token lifetime is 7 days. The `rest_framework_simplejwt.token_blacklist` app is installed.

- [x] Refresh token rotation is implemented

- [x] Refresh tokens have a maximum lifetime (7 days)
- [x] Refresh token revocation is supported (blacklist app)

### 908 - OIDC ID Token Not Validated
**Severity:** HIGH | **Result: PASS**

Google ID tokens are verified with signature validation, issuer check (`accounts.google.com` or `https://accounts.google.com`), audience check (client_id), and expiration check (handled by `jwt.decode`). Apple tokens are similarly verified against `https://appleid.apple.com` issuer with audience validation. The `algorithms=["RS256"]` restriction prevents alg:none attacks. Empty client_id is explicitly rejected to prevent audience bypass.

- [x] ID token signature is verified using the provider's public keys (JWKS)
- [x] Issuer (iss) claim is validated
- [x] Audience (aud) claim matches the client ID
- [x] Expiration (exp) claim is checked
- [x] Algorithm (alg: none) is rejected (explicit `algorithms=["RS256"]`)
- [x] Empty client_id rejected (security guard against audience bypass)

## Caching (926-950)

### 926 - Web Cache Deception
**Severity:** HIGH | **Result: PASS**

Backend API responses are JSON (not HTML) and served from `api.stepora.app`. Django REST Framework sets `Content-Type: application/json`. The CDN (CloudFront) is configured for the frontend SPA only, not API responses. API responses go through ALB which does not cache by default. The backend does not serve user-specific data on paths that could be confused with static resources.

- [x] API responses are JSON with appropriate content type
- [x] CDN is for frontend static assets only, not API
- [x] ALB does not cache API responses

---

### 927 - Web Cache Poisoning via Headers
**Severity:** HIGH | **Result: PASS**

The backend does not process X-Forwarded-Host, X-Forwarded-Scheme, or X-Original-URL headers (grep confirmed no usage). The only forwarded header used is `X-Forwarded-For` for IP detection and `X-Forwarded-Proto` for HTTPS detection. CloudFront serves static SPA files and does not cache dynamic API responses. The Vary header is managed by Django/DRF defaults.

- [x] Unnecessary header processing (X-Forwarded-Host etc.) is disabled
- [x] API responses are not served through CDN

---

### 928 - Web Cache Poisoning via Unkeyed Parameters
**Severity:** HIGH | **Result: N/A**

API responses are not cached by CDN or proxy caches. CloudFront only serves the frontend SPA static files. Backend responses go through ALB without caching.

---

### 929 - CDN Cache Poisoning
**Severity:** HIGH | **Result: PASS**

CloudFront is configured for the frontend SPA only (S3 origin with OAC). The CDN serves static files (JS, CSS, images) with content-hash filenames. Custom error responses (403/404 -> index.html) are configured for SPA routing. API traffic goes through ALB, bypassing CDN entirely.

- [x] CDN only serves static frontend assets
- [x] API traffic does not traverse CDN
- [x] Cache purge (CloudFront invalidation) is part of deployment

---

### 930 - Browser Cache Sensitive Data
**Severity:** MEDIUM | **Result: FAIL**

The backend does not explicitly set `Cache-Control: no-store` on API responses containing sensitive data. Django REST Framework does not add cache control headers by default. API responses with user profile data, subscription info, and other sensitive content may be cached by browsers.

- [ ] Cache-Control: no-store is NOT explicitly set on API responses -- **FAIL**
- [ ] Pragma: no-cache is NOT set -- **FAIL**
- [x] Refresh tokens are in httpOnly cookies (not in cacheable responses for web)

**Recommendation:** Add `Cache-Control: no-store, no-cache, must-revalidate, private` to all authenticated API responses via middleware.

---

- [ ] Cache-Control: private is NOT set on user-specific responses -- **FAIL**
